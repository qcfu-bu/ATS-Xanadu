## 1. Ruling (iv) — guarded `match` arms carry the guard in PyCore

### Contract change (`frontend/SATS/pycore.sats`)

`PCArm` gained an **optional guard** field (arity 3 → 4):

```
// before:  PCArm of (loctn, pcpat, pcexp)
and pcarm    = | PCArm of (loctn, pcpat, pcexpopt, pcexp)
and pcexpopt = | PCEGNone of () | PCEGSome of pcexp
```

`pcexpopt` follows the project's **monomorphic, non-dependent option** style (the same
dialect rule the PyAST `pyexpopt`/`pytypopt` use — M2-REPORT §2.6: trivially matchable,
transpiles cleanly; a dependent prelude `optn` would friction the standalone DATS). The
node-set comment was updated to record the ruling and that synthesized dispatch arms carry
`PCEGNone()`.

### Elaborator change (`frontend/DATS/pyelab_core.dats`, `el_arms`)

The old code desugared a guarded surface arm to an inner `if`:

```
| PyExpSome(g) => PCEif(loc, el_exp(g), pc_body0, pc_body0)   // WRONG: no fall-through
```

That is semantically wrong — a failed guard must fall through to the **next** arm
(ML/ATS `case` semantics), which an inner `if` (whose then/else are the *same* body)
cannot express. The new code **elaborates the guard and attaches it to the arm**:

```
val pc_g = (case+ gopt of PyExpNone() => PCEGNone() | PyExpSome(g) => PCEGSome(el_exp(g)))
... PCArm(loc, pc_p, pc_g, pc_body)
```

M4 will lower a guarded arm to ATS's native guarded clause (`d2cls` `D2GPTgua`), which has
the correct fall-through.

### Propagation (every `PCArm` consumer + producer)

Changing `PCArm`'s arity breaks every site that matches/builds it. Updated:

| File | Site | Change |
|---|---|---|
| `pyelab_core.dats` | `el_arms` (surface arms) | preserve the guard (above) |
| `pyelab_core.dats` | `el_func_body` epilogue arms | add `PCEGNone()` to the 2 synthesized `flow_return`/`flow_next` arms |
| `pyelab_loop.dats` | `mk_arm_con`, `mk_arm_ret`, `a_done`, `a_more` | add `PCEGNone()` to all synthesized dispatch arms |
| `pyelab_print.dats` | `pp_armlst` | print the preserved guard as `... if <g> => ...` (golden proof) |
| `pyelab_lint.dats` | `lint_arms` / `lint_loops_arms` | walk the guard in **scrutinee (non-tail)** position |
| `pyelab_diag.dats` | `harv_arms` / `uses_arms` | harvest poison nodes + scan pyrt names inside the guard |

**M3 propagation:** M3's `pylower_dynexp.dats` has `PCEcase(loc, _, _) => d2exp_none0(loc)`
as a deferred placeholder — it does **not** destructure `PCArm`, so the arity change does
not break it; `build-m3.sh` still builds + passes. (`PCEcase`'s own arity is unchanged.)
A grep confirmed no other M3 (`pylower_*`) file matches `PCArm`.

## 2. Ruling (i) — threaded / pure iterator for `for`

### What was wrong

The old `for_iter_loop` (`pyelab_loop.dats`) emitted a **stateful** shape: `it` was bound
once by `let val it = iter_open(...)` and `iter_step(it)` was re-applied to the *same* `it`
every iteration (the `iter_more` pattern bound only the element, no advanced iterator). The
loop relied on an implicit side-effecting iterator — not re-entrant, and at odds with the
§7.1 capture-by-value purity guarantee.

### The threaded shape now emitted

The iterator state `it` is **threaded through the loop's accumulator tuple** — it becomes
the loop's first parameter, and the advanced iterator `it'` (bound from `iter_more(x, it')`)
is passed as the first element of the tail self-call's tuple:

```
let
  fun loop((it, <accs>)) =
    case iter_step(it) of
    | iter_done()        => flow_next(<accs>)              // it never escapes the flow
    | iter_more(x, it1)  =>
        case <body-flow over accs> of
        | flow_next(accs')  => loop((it1, accs'))          // TAIL — threads it1
        | flow_cont(accs')  => loop((it1, accs'))          // TAIL — threads it1
        | flow_break(accs') => flow_break(accs')
        | flow_return(r)    => flow_return(r)
in
  <outer dispatch> loop((iter_open(iter), <accs0>))
```

`it` is an **ordinary immutable value** carried as a tail-call argument — there is **no
`var`/`ref` cell**, the pass stays re-entrant, and the self-call stays in **tail position**,
so §6 holds (the backend compiles it to a real `while`). The iterator state is internal to
the loop: the loop's *result flow* (`flow_next`/`flow_break`/`flow_return`) carries only the
user accumulators, never `it`.

### Implementation (`pyelab_loop.dats`)

New standalone helpers placed **before** the mutually-recursive worker group (so they do not
break the `and`-chain — M2 Δ3 dialect care):

- `it_params(accs) = it :: accs` — the loop's parameter nameset (iterator state prepended).
- `accs_var_exps(loc, accs)` — accs names → `(Evar nm)` list for splicing.
- `loop_call_it(loc, accs)` — the tail self-call `loop((it1, accs'))`.
- `loop_call_init(loc, openexp, accs)` — the init call `loop((iter_open(iter), accs0))`,
  matching `accs_tuple_exp`'s single-vs-tuple convention on the `it :: accs` shape.

`for_body_dispatch` now uses `loop_call_it` on the `flow_next`/`flow_cont` arms;
`for_iter_loop` binds `iter_more(x, it1)` (two-field constructor pattern) and declares the
loop with `it_params(accs)`, dropping the old outer `let val it = iter_open(...)`.

### `pyrt` protocol — already threaded, comment updated

`frontend/pyrt/pyrt.dats` **already specified the threaded contract** (the M2.5 agent wrote
the threaded `iterstep`/`iter_*` signatures but the elaborator emitted the stateful shape):

```
datatype iterstep(x:t0) = iter_done of () | iter_more of (x, list(x))   // (next, remaining)
fun <x:t0> iter_open_list(xs: list(x)): list(x)        // v1 container #1 = lists
fun <x:t0> iter_step_list(it: list(x)): iterstep(x)
fun range(lo: sint, hi: sint): list(sint)              // v1 container #2 = integer range
```

So **no `pyrt` redesign was needed** — only a comment update marking the
stateful-vs-threaded item RESOLVED (the elaborator now matches the signature exactly).
v1 containers: **lists** (`it` = the list; step = cons/nil) and **`range`** (`it` = the
current index, materialized to a list). The change is comment-only (verified: `git diff`
of `pyrt.dats` touches only `//` lines).

**Discrepancy found (pre-existing, NOT introduced, out of scope):** a control-pure
single-accumulator `for` with a `break` in its body is mis-routed to the `list_foldleft`
fast path (`for_value`'s `single && noelse` test ignores body-level control flow), where the
`break` then poisons to `"break outside a loop"`. This is latent in `for_value`'s fast-path
selection (unchanged by this refinement) — it decides fast-vs-iterator on accumulator
*arity*, not on whether the body is control-bearing. The threaded-`for` golden
(`m25_for_range`) uses a two-accumulator control-pure body, which correctly routes through
`for_iter_loop`; the guarded `m25_for_else`/`m25_nested_return` goldens (control-bearing →
flow mode → `for_iter_loop`) also exercise the threaded path. **Recommend** a follow-up:
route a `for` to `for_iter_loop` whenever its body's `flags_stmts` are control-bearing,
independent of accumulator arity (additive; would also let `for … break` with one
accumulator work).
