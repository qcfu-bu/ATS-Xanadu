## 2. The PyCore IR design + rationale (the M3 contract)

PyCore (`frontend/SATS/pycore.sats`) is a **functional-only** core (LOOP-DESUGARING §2): by
the time it exists, **all imperative control is eliminated** — there is no `mut`, loop,
`break`, `continue`, or `return` node left for M3 to desugar; M3 is a straight structural map.

**Node set** (each carries a real `loctn`; `loctn_dummy()` only for the synthesized `loop`
binder — §9):

- `pcexp`: `PCElit` · `PCEvar`(LIDENT) · `PCEcon`(UIDENT) · `PCEapp` · **`PCElam`
  (EXPRESSION body, not a stmt suite — §2)** · **`PCElet`** (immutable, the SSA-rebind
  workhorse) · **`PCEletfun`** (recursive `fun` group — the generated `loop` and surface
  defs) · `PCEif` (3-branch) · `PCEcase` · `PCEtup` · `PCErec` · `PCElist` · `PCEfield` ·
  `PCEseq` (side-effect sequencing → L2 `D2Eseqn`) · `PCEunit` · `PCEerror` (poison).
- `pcpat`: `PCPvar` · `PCPwild` · `PCPcon` · `PCPtup` (destructure the accumulator tuple at
  loop boundaries) · `PCPrec` · `PCPlit`.
- `pcdecl`: `PCCdata` (datatype, carried through) · `PCCfun` (a `fun` group from `def`s) ·
  `PCCval` (top-level binding/effect) · **`PCCstaload`** (the `staload pyrt` the elaborator
  emits when the module used the machinery) · `PCCerror`.
- `pcmodule = PCModule(decls, diags)`; `pcdiag = PCDiag(loc, msg)`.

**How loops/mut/control became functional core:**

- `let mut` → an ordinary immutable `PCElet` (mutability was a *binding-class* fact the §4
  analysis used; the core binding is immutable).
- reassignment `x = e` → an SSA shadowing `PCElet` re-binding the same name (later reads see
  the new value); a reassign of a non-mut → a `PCEerror` poison + diagnostic.
- `while`/`for` → a recursive `PCEletfun` whose `loop` self-call is in **tail position**
  (§6); the backend then loops it. The `loop` name is bound before its body so the self-call
  resolves to the same `d2var` the backend's tail test keys on (§9).
- `break`/`continue`/`return` → applications of the pyrt `flow` constructors threaded by the
  inlined `flow_bind`; **in control-pure suites they never appear** (§3.1 fast path).

**Why `flow`/iterator/`foldleft` are NOT PyCore nodes** (load-bearing): they are ordinary
con/var references (`PCEcon "flow_next"`, `PCEvar "flow_bind"`, …) into `pyrt`, resolved by
M3 through the ordinary `tr12env` fall-through (§9). Keeping them out of the node set is
exactly what lets the §8 join-point optimization swap the `flow` *representation* later
without touching PyCore or M3. The **UIDENT/LIDENT split is preserved** (`PCEcon` vs
`PCEvar`, `PCPcon` vs `PCPvar`) so M3 reads con-vs-var off the node kind without
re-resolving names — same contract discipline as the PyAST (M2-REPORT §2.2).

## 3. The accumulator analysis + elaboration approach

**§4 accumulator-set analysis** (`pyelab_core.dats`): a bottom-up `assigned_stmts` returns
the set of names reassigned (`PySreassign` to an LIDENT) anywhere in a suite — including
nested `if`s and nested loops — but **stops at inner `def`/`lambda`** (§7.1: a reassignment
inside an inner function is that function's business, not the loop's accumulator). At each
loop site `A(loop) = muts_in_scope ∩ assigned(body ∪ else)`, in **declaration order** (the
`nameset` is an ordered list with order-preserving union/intersect, so the tuple order is
stable for codegen). `let mut x` *declarations* do not count as assignments; a `mut`
declared inside the loop body is iteration-local and excluded by the intersect.

**§3.1 two modes** (`flags_stmts` computes `@(may_return, may_break, may_continue)`; a nested
loop **consumes** its own break/continue, only `return` propagates up):

- **control-pure suite → the allocation-free fast path** (`el_pure`): a straight chain of
  `PCElet`s ending in the tail value; reassignments are SSA `PCElet` rebinds; a control-pure
  `while` degrades to the §10.1 shape — **no `flow`, no `case`** (verified: the 10.1 dump
  has zero `flow` ctors and no `staload pyrt`). A control-pure single-accumulator `for` with
  no `else` collapses to a `list_foldleft` (the §5.3 fast path).
- **control-bearing suite → flow mode** (`fl_suite`): sequencing is the inlined `flow_bind`
  `case` (only `flow_next` runs the continuation; the three signals short-circuit);
  fall-off-the-end is `flow_next(accs)`; `break`/`continue`/`return` build the pyrt ctors.

**§5.2 `while` / §5.3 `for` combinators** (`pyelab_loop.dats`): the inner dispatch maps
`flow_next`/`flow_cont` → `loop(accs')` (**TAIL**), `flow_break` → `flow_break(accs')`,
`flow_return` → `flow_return(r)`; the outer dispatch reframes the loop's own break/`else`
and lets `flow_return` pass straight through (so an inner loop's `break` never touches an
outer loop, and an early `return` threads through *both* loops up to the §5.4 epilogue —
verified in 10.3). The loop-`else` runs on the `flow_next` exit, is skipped on `flow_break`
(§7.3, verified in 10.4). The §5.4 **function epilogue**: a control-pure body is just its
tail value; a return-bearing body is `case <body-flow> of flow_return(r) => r |
flow_next(_) => tailValue`.

**Diagnostics** (non-fail-fast, total): a misuse emits a `PCEerror`/`PCCerror` poison node
inline + keeps elaborating; `harvest_decls` then walks the PyCore collecting every poison
message into the module `pcdiaglst` (so the elaborator's signatures stay clean
`pyexp→pcexp`). Covered: reassign-immutable (10.x diag case), `break`/`continue` outside a
loop (diag case), `return` outside a function. The elaborator is **pure / re-entrant**:
`pyelab_module(ast)` returns a fresh `pcmodule`; no global state.

## 4. The §6 tail-position lint

`pyelab_lint.dats` asserts the load-bearing invariant at elaboration time. For every
`PCFundcl` with `isloop=true` it walks the body with a `tail: bool` flag (if-branches tail,
cond not; case-arm bodies tail, scrutinee not; let-body tail, rhs not; seq-second tail,
first not; app args/head not tail) and flags any `loop(...)` application in non-tail
position. **A `TAIL-LINT VIOLATION` diagnostic is a hard build failure** in `build-m2_5.sh`.

> **Subtlety found + fixed:** a nested loop also names its function `loop`. The lint must
> **stop at any nested `Eletfun` that rebinds `loop`** — inside it, `loop` refers to the
> *inner* loop (independently linted via `lint_loops_fundcls`), so descending would
> mis-flag the inner loop's legitimate tail self-calls (and its non-tail outer-dispatch
> `case loop(...)`) as the outer loop's violations. With the `binds_loop` guard, the nested
> 10.3 case lints **clean**.

Verified: all generated loops in 10.1/10.2/10.3/10.4 pass; the lint catches the planted
nested-loop scenario only when genuinely non-tail.

`frontend/pyrt/pyrt.dats` (type-checks `nerror=0`). Proposed v1 protocol:


**v1 minimum = lists + an integer `range`** (per the task). Decisions flagged for your ruling:

1. **Threaded vs stateful iterator.** I propose the **threaded** form: `iter_step` returns
   `iter_more(x, it')` with the *advanced* iterator, threaded by the loop (no hidden cell →
   preserves §7.1 capture-by-value + re-entrancy). **However**, the elaborator currently
   *emits* the *stateful* reading (`let val it = iter_open(...)` once, then `iter_step(it)`

   threaded `it'` (purer) or the stateful `it` (fewer allocations)? Until you rule, the
   PyCore dump shows the stateful shape; pyrt documents the threaded contract. *Single-line

2. **Container set.** v1 = lists + range only. Strings/dicts/sets are post-v1 and **additive**

3. **`flow_bind` is generic but unused by the emitted PyCore** (the elaborator *inlines* the

## 7. Conformance to the §7 semantic decisions

- **§7.1 closures capture by value** — falls out of the functional model: a lambda inside a
  loop captures the current SSA value (the `PCElet`-bound name), not a live cell; §4 stops
  the assigned-set scan at inner-function boundaries. ✓
- **§7.2 break/continue → innermost loop; return → function** — enforced by the combinator
  structure: the nearest loop's dispatch consumes `flow_break`/`flow_cont`; `flow_return`
  passes through every loop boundary (verified in 10.3, return threads through both loops). ✓
- **§7.3 loop-`else`** — runs on the `flow_next` exit, skipped on `flow_break` (verified in
  10.4); the `else` is elaborated with the enclosing `muts` in scope (a fix — without it, a
  reassignment in the `else` was falsely flagged non-mut). ✓
- **§7.4 definite-assignment** — partial (target validation only); full check flagged above.
